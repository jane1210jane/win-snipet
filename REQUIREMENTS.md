# Windows 11用スニペットツール 要件定義書

## 1. 目的

Windows 11上で常駐し、ユーザが登録した文字列を、割り当てたショートカットキーによって現在操作中のアプリケーションへ貼り付ける個人向けツールを提供する。

## 2. 対象範囲

### 2.1 対象

- スニペットの登録、表示、編集、削除
- スニペットごとのショートカットキー設定
- ショートカットキーによる貼り付け
- GUIによる全設定
- Windowsサインイン時の自動起動
- タスクトレイでの常駐
- 多重起動の防止

### 2.2 対象外（初期リリース）

- 複数PC間の同期、クラウド保存
- 複数ユーザ間での設定共有
- 画像、ファイル、リッチテキストの貼り付け
- 変数展開、マクロ、スクリプト実行
- Windows 11以外のOS対応
- 管理者権限を必要とするアプリへの入力保証

## 3. 利用条件

- 対象OSは、Microsoftのサポート対象である64-bit版Windows 11とする。
- 個人ユーザが、自身のWindowsアカウントで利用する。
- 通常ユーザ権限でインストール、起動、設定、利用できることを原則とする。
- オフラインで主要機能を利用できること。

## 4. 用語

- **スニペット**: ユーザが登録する、貼り付け対象のプレーンテキスト。
- **ショートカットキー**: スニペットを呼び出すグローバルなキーの組み合わせ。
- **設定画面**: スニペットおよびアプリ動作を設定するGUI。
- **常駐**: GUIを閉じてもバックグラウンドでショートカットキーを待ち受ける状態。

## 5. 機能要件

### FR-01 常駐動作

- アプリは1プロセスで常駐すること。
- 設定画面を閉じても終了せず、タスクトレイに常駐すること。
- タスクトレイから設定画面の表示とアプリの終了ができること。
- 常駐中はポーリングを避け、Windowsのイベント通知を利用すること。
- 補助プロセスや常駐サービスを起動しないこと。
- スレッド数は採用するGUI/runtimeが必要とする最小限とし、アプリ独自の常時稼働ワーカースレッドを原則作成しないこと。

### FR-02 スニペット管理

- GUIからスニペットを新規登録、表示、編集、削除できること。
- 各スニペットは少なくとも次の項目を持つこと。
  - 識別用の名称
  - 貼り付ける本文
  - ショートカットキー
  - 有効／無効
- 本文はUnicodeおよび複数行を扱えること。
- 名称、本文、ショートカットキーの入力値を保存前に検証し、不備をGUI上に表示すること。
- 削除時は確認を求めること。

### FR-03 ショートカットキー

- アプリが非アクティブでも、有効なスニペットのショートカットキーを検知できること。
- ショートカットキーは修飾キー（Ctrl、Alt、Shift、Windowsキーのいずれか）と通常キーの組み合わせを基本とすること。
- 同一ショートカットキーを複数スニペットへ割り当てできないこと。
- OSまたは他アプリとの競合により登録できない場合、対象キーと理由をユーザへ表示すること。
- 設定変更後、再起動せずに割り当てを反映すること。

### FR-04 貼り付け

- ショートカットキーを押した時点でフォーカスを持つ入力先へ、対応する本文を1回貼り付けること。
- 日本語、英数字、記号、改行を含むプレーンテキストを扱えること。
- 貼り付け失敗時にアプリが停止しないこと。
- クリップボードを利用する場合、貼り付け前の内容を可能な範囲で復元し、復元不能または競合時の動作を仕様化すること。
- パスワード等の機密入力欄へ貼り付けるかどうかは、実装前に決定すること。

### FR-05 設定GUI

- すべてのユーザ設定をGUIから変更でき、設定ファイルの直接編集を必須としないこと。
- 一覧画面で名称、ショートカットキー、有効状態を確認できること。
- 登録・編集画面で本文とショートカットキーを入力できること。
- スタートアップ起動の有効／無効を変更できること。
- ウィンドウを閉じた場合の動作が「常駐継続」であることを明示すること。

### FR-06 設定保存

- 設定は実行ファイルと同じフォルダーへ保存すること。
- 設定はアプリ終了およびWindows再起動後も保持されること。
- 保存処理中の異常終了で既存設定が全損しないよう、置換可能な一時ファイル等を用いて安全に更新すること。
- 設定形式にバージョンを持たせ、将来の移行を可能にすること。

### FR-07 多重起動防止

- 同一Windowsユーザセッション内で起動できる常駐インスタンスは1つだけとすること。
- OSの名前付きMutex等、競合を原子的に判定できる仕組みを使用すること。
- 2つ目の起動要求を検知した場合、新しい常駐プロセスを残さず、既存インスタンスの設定画面を前面に表示して終了すること。
- 異常終了後に、不要なロックが永続して再起動不能にならないこと。

### FR-08 Windowsスタートアップ

- 初回正常起動時に、現在のWindowsユーザのスタートアップへ自動登録すること。
- 登録に管理者権限を要求しないこと。
- GUIから自動起動を無効化・再有効化できること。
- 実行ファイルの場所が変わった場合、次回手動起動時に登録先を現在のパスへ修復できること。
- アンインストール時に、本アプリが作成したスタートアップ登録を削除すること。
- 登録方式は実装設計時に確定する。候補はユーザ単位の `HKCU\\Software\\Microsoft\\Windows\\CurrentVersion\\Run` とする。

### FR-09 終了・障害処理

- タスクトレイの終了操作で、ショートカットキー登録を解除して正常終了すること。
- 設定読み込みに失敗した場合、原因を表示し、破損データを上書きせずに設定画面を開けること。
- ユーザが確認できるエラー情報を提供し、通常利用時にコンソール画面を表示しないこと。

## 6. 非機能要件

### NFR-01 軽量性

- 定常時は1プロセスとし、子プロセスおよびWindowsサービスを使用しないこと。
- 待機中に継続的なCPUポーリングを行わず、CPU使用率は一般的なアイドル環境で継続的に観測されない水準を目標とすること。
- メモリ使用量とスレッド数は、技術選定時に候補ごとに実測し、採用判断の資料へ記載すること。
- 起動性能、メモリ上限、スレッド数上限の具体値は、実装方式の比較計測後に受入基準として確定すること。

### NFR-02 信頼性

- 1回のショートカット操作で重複貼り付けを発生させないこと。
- 不正な設定値や競合キーがあっても、他の有効なスニペットは利用可能であること。
- Explorerの再起動後、タスクトレイアイコンを復元できること。

### NFR-03 セキュリティ・プライバシー

- スニペット本文を外部へ送信しないこと。
- テレメトリや利用状況送信を初期リリースでは実装しないこと。
- ログへスニペット本文やクリップボード内容を記録しないこと。
- 設定ファイルは実行ファイルと同じフォルダーへ平文保存し、暗号化保管は保証しない。

### NFR-04 保守性・テスト

- TDDを基本とすること。
- 静的解析、ユニットテスト、結合テスト、E2Eテストを実施すること。
- Windows依存処理（ホットキー、Mutex、スタートアップ、クリップボード）とドメイン処理を分離し、可能な範囲を自動テスト可能にすること。
- GitHub Issue単位でブランチを作成し、プルリクエストでレビュー後にマージすること。

### NFR-05 配布

- GUIアプリケーションとして単一の起動用 `.exe` を提供することを基本とする。
- 利用者環境へ開発環境の導入を要求しないこと。
- 配布物、設定、ログの配置先を設計書に記載すること。
- コード署名の有無およびインストーラ形式は別途決定すること。

## 7. 外部ライブラリ・runtimeの承認

- `.exe` 化または実行のために、OS標準機能および採用言語の標準ライブラリ以外のライブラリ、runtime、パッケージ、バンドラ、インストーラ作成ツールを追加する場合、導入前にユーザの明示的な承認を得ること。
- 承認依頼には最低限、名称、バージョン、用途、ライセンス、配布物への同梱有無、概算サイズ、代替案、更新・脆弱性対応方針を記載すること。
- 承認前は、対象依存物のインストール、プロジェクトへの追加、ロックファイル更新、配布物への同梱を行わないこと。
- Windows標準APIのみで実現可能な箇所は、外部依存を増やさない方式を優先して比較すること。

## 8. 受入条件（初期案）

- 10件以上のスニペットをGUIで登録、編集、削除でき、再起動後も保持される。
- 任意の3種類以上の一般的なデスクトップアプリで、日本語と複数行を含む本文をショートカットから貼り付けられる。
- 同一キーの重複、OS等とのキー競合を検出し、GUIへ表示できる。
- 同時に複数回起動しても常駐プロセスが1つだけであり、後続起動で既存GUIが表示される。
- Windowsへサインインし直した後、自動で1インスタンスだけ起動する。
- 自動起動をGUIで無効化した場合、次回サインイン時に起動しない。
- 待機中に子プロセス、サービス、アプリ独自のポーリング処理が存在しない。
- 異常な設定ファイルを与えてもクラッシュや既存データの無条件上書きをせず、エラーを確認できる。
- 承認されていない外部依存物が配布物に含まれていない。

## 9. 実装前に確定が必要な事項

1. 1スニペットにつき1つのグローバルショートカットを割り当てる仕様でよいか。
2. 利用を許可するキー範囲と、予約・禁止するキーの組み合わせ。
3. クリップボードの退避・復元方針と、他アプリが同時更新した場合の扱い。
4. パスワード欄等の機密入力欄への貼り付けを許可するか。
5. スニペット本文を平文保存するか、Windows DPAPI等で暗号化するか。
6. 初回起動時に無条件でスタートアップ登録するか、初回GUIで通知して登録するか。
7. ポータブル版、インストーラ版、または両方を提供するか。
8. コード署名を行うか。
9. 軽量性の定量基準（アイドル時メモリ、スレッド数、起動時間）。
10. 対象とする入力先アプリと、管理者権限で動作するアプリへの対応要否。

## 10. 要件トレーサビリティ方針

- GitHub Issueには関連する要件ID（例: `FR-07`）を記載すること。
- テストケースには検証対象の要件IDを記載すること。
- 要件変更時は、関連Issue、設計、テストへの影響を確認すること。
